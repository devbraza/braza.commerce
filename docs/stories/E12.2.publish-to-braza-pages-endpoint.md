## Story

Como sistema, preciso de um endpoint que renderize o HTML da pagina e envie para o braza.pages via BrazaPagesService, para que o frontend possa disparar a publicacao com um unico request.

## Acceptance Criteria

- [ ] Novo endpoint: `POST /pages/:id/publish-to-braza-pages`
- [ ] Valida que a pagina existe e tem status PUBLISHED
- [ ] Renderiza o HTML completo via `RenderService.render(page)` — mesmo HTML que o `GET /p/:slug` serve
- [ ] Inclui imagens no HTML (mesmo carousel do render existente)
- [ ] Envia HTML + slug + domain_id (da env `BRAZA_PAGES_DEFAULT_DOMAIN_ID`) para `BrazaPagesService.publish()`
- [ ] Em caso de sucesso: salva `brazaPagesUrl`, `brazaPagesDeployId` e `brazaPagesPublishedAt` no model Page
- [ ] Retorna `{ success: true, url: "https://...", deploymentId: "..." }` ao frontend
- [ ] Se pagina nao esta PUBLISHED: retorna 400 com mensagem clara
- [ ] Se braza.pages retorna erro: retorna o codigo e mensagem do braza.pages ao frontend
- [ ] Se env vars nao configuradas: retorna 503 com mensagem "Integracao braza.pages nao configurada"
- [ ] Zero alteracoes no endpoint existente `PATCH /pages/:id/publish` — sao independentes
- [ ] HTML enviado e o render BASE (sem campaign tracking) — `renderService.render(page)` sem parametros de campaign

## Dev Notes

- Este endpoint reutiliza o `RenderService` existente — NAO cria um render novo
- **RenderService injection:** O `RenderService` vive no `RenderModule` (ou no `PublicModule`). Para usar no `PagesController`, importar o modulo que exporta o `RenderService` no `PagesModule`. Verificar onde o `RenderService` e declarado como provider/export e importar esse modulo.
- O endpoint existente `PATCH /pages/:id/publish` continua funcionando independente
- `domain_id` vem da env var no MVP — frontend nao envia domain_id
- Para buscar as imagens da pagina: `page.images` (relacao Prisma)
- **Campaign tracking:** O HTML enviado ao braza.pages e SEMPRE renderizado SEM campaign (`renderService.render(page)` sem parametros de campaign). Isto significa que o HTML no braza.pages e o render BASE, sem tracking script. Quando ha campaign ativa, o `GET /p/:slug` do braza.commerce serve uma versao diferente (com tracking). Ambas as versoes coexistem — sao complementares, nao conflitantes.
- O HTML inclui as imagens como URLs absolutas apontando para o backend do braza.commerce (CON-01 do PRD)
- INTOCAVEIS: `template.html`, `render.service.ts`, `public.controller.ts`
