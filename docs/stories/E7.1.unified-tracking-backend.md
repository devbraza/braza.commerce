> **Epic:** E7 - Fluxo Unificado (pagina + tracking)

> **Agente:** @dev (Dex)

## Story

Como usuario, ao criar uma landing page quero informar a URL do checkout, Pixel ID e Access Token no mesmo fluxo, sem precisar criar uma campanha separada.

## Contexto Tecnico

A arquitetura define **Opcao B**: manter a tabela Campaign existente, mas tornar transparente. O backend cria Campaign automaticamente quando o usuario informa campos de tracking. Pro usuario, parece que Pagina = Campanha.

**JA EXISTE e NAO deve ser alterado:**
- Model Campaign, Click, Event no Prisma
- TrackingService (registro de clicks e eventos)
- CampaignsService (CRUD + stats)
- CapiService (Meta Conversion API)
- RenderService (injeta script de tracking)
- Script de tracking no template HTML

## Acceptance Criteria

- [ ] `CreatePageDto` aceita campos opcionais: `checkoutUrl`, `pixelId`, `accessToken`
- [ ] `UpdatePageDto` aceita campos opcionais: `checkoutUrl`, `pixelId`, `accessToken`
- [ ] `PagesService.create()`: se `checkoutUrl` informada, cria Campaign automatica (name = page.title, status = ACTIVE)
- [ ] `PagesService.update()`: se campos de tracking informados, cria ou atualiza Campaign da pagina
- [ ] `CampaignsService`: novo metodo `findActiveByPageId(pageId)` que retorna a Campaign ativa da pagina
- [ ] `PagesController`: novo endpoint `GET /pages/:id/stats?from=&to=` que busca Campaign ativa e retorna stats via `CampaignsService.getStats()`
- [ ] `GET /pages` (findAll): retorna campos de tracking (checkoutUrl, pixelId) da Campaign ativa junto com cada pagina
- [ ] `PagesService.publish()`: se Campaign ativa existe, passa campaignId pro `RenderService.render()` (script tracking embutido no HTML)
- [ ] Se nao tem Campaign (campos de tracking nao informados), pagina publica normalmente sem tracking
- [ ] ~~Rate limiting~~ → movido pra E8.1
- [ ] ~~Idempotencia webhook~~ → movido pra E8.1
