# Story E8.2 — CAPI Event Deduplication (event_id)

> **Epic:** E8 - Pixel Server-Side
> **Status:** Ready
> **Prioridade:** MUST
> **Agente:** @dev (Dex)
> **Depende de:** E8.1
> **Origem:** Bug report — conversoes duplicadas no Facebook apos integracao Yampi

## Contexto

Apos integrar a API da Yampi, conversoes de Purchase apareciam duplicadas no Facebook. A causa principal era o Pixel da Yampi (browser) + CAPI do braza.commerce (server-side) disparando o mesmo evento. O Pixel da Yampi foi desativado em 2026-03-28.

Esta story adiciona `event_id` ao payload do CAPI como protecao contra duplicacao por retry de webhook ou falha de rede.

## Story

Como sistema, preciso enviar um `event_id` unico em cada evento do CAPI para que o Facebook possa deduplicar eventos recebidos mais de uma vez na janela de 48h.

## Acceptance Criteria

- [ ] Cada chamada CAPI inclui campo `event_id` unico no payload
- [ ] `event_id` segue formato: `{event_type}_{clickId}_{timestamp}`
- [ ] Deduplicacao verificavel no Events Manager do Facebook (aba "Test Events")
- [ ] Nenhuma regressao nos eventos ViewContent e Purchase existentes

## Escopo

**IN:**
- Adicionar `event_id` no payload de `sendViewContent()` e `sendPurchase()`
- Gerar ID deterministico baseado no click + tipo de evento

**OUT:**
- Alteracao no schema do banco (nao necessario — event_id e gerado on-the-fly)
- Retry logic para CAPI (escopo de outra story)
- Client-side pixel (nao usamos)

## Dev Notes

- Meta requer `event_id` como string unica por evento dentro de janela de 48h
- Formato sugerido: `vc_{click.clickId}_{timestamp}` e `purchase_{click.clickId}_{timestamp}`
- Alternativa mais simples: usar o `click.id` (cuid) + tipo — ja e unico por natureza
- Referencia Meta: https://developers.facebook.com/docs/marketing-api/conversions-api/parameters
